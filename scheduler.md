# TaskPlugin 调度方案设计建议

## 1. 问题背景

当前项目已经具备两种 Spark 任务提交方式：

1. 单任务单应用模式
- 每次提交一个任务目录。
- 每次 `spark-submit` 对应一个独立 Spark Application。

2. 单应用多任务模式
- 一次提交一个宿主应用。
- 宿主应用内部通过 `TaskManager + TaskRunner` 并发调度多个 `sql.yaml`。

现阶段的问题不是“能不能提交”，而是“谁来负责自动化调度、调度边界放在哪里更合适”。

核心决策点是：
- 是在当前项目内部把两种调度逻辑都做完整；
- 还是单独再做一个外部调度框架项目；
- 或者采用内外分层的混合方案。

## 2. 三种可选方案

### 2.1 方案 A：全部放在当前项目内部

思路：
- 在当前项目里同时实现：
  - 单任务单应用模式的自动调度；
  - 单应用多任务模式的自动调度；
  - 定时触发、依赖控制、失败重试、运行记录等能力。

优点：
- 所有逻辑都在一个项目里，开发初期看起来集中。
- 不需要额外维护外部调度服务。
- 对小规模、单团队试验场景比较直接。

问题：
- 当前项目本质上是“计算执行器”，不是“控制平面”。
- 一旦把触发、编排、状态管理、重试、限流、任务依赖都塞进来，项目职责会迅速膨胀。
- 两种 Spark 提交方式会形成两套较重的调度分支，后续维护成本高。
- 如果后面还要接入 Flink、Python、Shell 等任务类型，这种设计会越来越难扩展。

结论：
- 不建议作为长期方案。
- 只适合很短期、很小规模的内部验证。

### 2.2 方案 B：完全独立一个外部调度框架项目

思路：
- 当前项目只负责“被调用后执行任务”。
- 新建一个专门的调度框架项目，负责：
  - 定时触发；
  - 任务编排；
  - 依赖管理；
  - 重试与告警；
  - 按规则选择调用“单任务模式”还是“多任务模式”。

优点：
- 职责清晰：执行归执行，调度归调度。
- 更容易演进为统一控制平面。
- 更适合后续扩展到多引擎、多类型任务。

问题：
- 如果把所有调度逻辑都移到外部，外部项目必须理解很多 Spark 内部执行细节。
- 单应用多任务模式本来就天然需要“应用内调度”：
  - 子任务队列；
  - 并发控制；
  - 生命周期治理；
  - 临时对象隔离；
  - 宿主应用何时退出。
- 这些逻辑如果强行完全放到外部，会导致边界不清，甚至重复造轮子。

结论：
- 方向是对的，但“完全外置”并不是最优。
- 外部项目适合做“总调度”，不适合替代应用内部的执行期调度。

### 2.3 方案 C：混合方案

思路：
- 外部调度项目负责“总调度/控制面”。
- 当前项目内部保留“执行期轻量调度”。

职责划分：

外部调度项目负责：
- 定时触发；
- 任务编排和依赖；
- 决定用哪种提交模式；
- 生成提交参数；
- 提交到 YARN；
- 记录运行实例、状态、重试、告警。

当前项目内部负责：
- 单应用多任务模式下的子任务发现；
- 子任务并发控制；
- 任务级失败隔离；
- 生命周期治理；
- 临时对象命名隔离；
- 应用内状态汇总。

优点：
- 职责边界最清晰。
- 复用当前项目已经实现的多任务宿主能力，不重复实现内部队列调度。
- 外部项目不需要深入侵入 Spark 执行细节，只做模式选择和任务编排。
- 后续如果新增 Flink 或其他执行器，外部调度层可以统一扩展。

问题：
- 需要定义清楚“外部调度”和“内部调度”的边界与接口。
- 需要额外维护一个外部调度项目。

结论：
- 这是当前最合适的方案。

## 3. 推荐方案

我建议采用方案 C：外部总调度 + 项目内轻量调度。

原因很直接：

1. 单应用多任务模式本身就要求应用内调度
- 这不是可选项，而是执行模型的一部分。
- `TaskManager`、`TaskRunner`、生命周期治理这些逻辑天然应该留在当前项目里。

2. 单任务单应用模式和单应用多任务模式的“触发与编排”应统一放到外部
- 哪些任务该什么时候跑；
- 哪些任务应该合批到同一个宿主应用；
- 哪些任务应该单独提交；
- 失败后是否重试、是否告警；
- 这些都属于控制平面，而不是执行器本身。

3. 可以避免两边都写重
- 不要在当前项目里再做一套完整的“全局调度系统”。
- 也不要在外部项目里重写一套“应用内队列调度器”。
- 两边各做自己最擅长的一层。

## 4. 业务背景下的目标调度模型

结合当前项目背景，未来需要承接的 Spark 计算业务可以明确分为两类。

### 4.1 第一类：单进程多线程的宿主式应用

这类业务的特点是：
- 调度系统按固定周期拉起一个 Spark Application；
- 启动时指定一个 `app-config.yaml`；
- 该配置文件指向一批 `sql.yaml` 所在目录；
- 当前 Spark Application 作为宿主进程，在内部发现并执行这一批 SQL 任务；
- 这些任务在同一个 SparkSession 中，以子线程/任务队列方式完成调度与执行。

这类业务适合：
- 一批小任务统一处理；
- 希望减少 Spark Application 数量；
- 希望复用同一个 SparkSession 和 Executor 资源；
- 同一业务批次内的 SQL 任务彼此相对独立，但属于同一类计算主题。

### 4.2 第二类：多进程的编排式应用

这类业务的特点是：
- 每个进程的计算逻辑不同；
- 每个进程对应自己独立的 `config.yaml + sql.yaml`；
- 不同进程之间可能有前后依赖；
- 调度系统需要识别依赖关系，按顺序或按 DAG 编排提交多个 Spark Application；
- 在 `spark-submit` 时，还可能需要给每个进程注入运行参数，使实际执行内容可以随调度批次变化。

这类业务适合：
- 单个任务资源占用较大；
- 任务之间存在明显依赖链路；
- 不同任务的提交参数、资源规格、执行窗口差异较大；
- 希望通过独立 Spark Application 获得更强隔离性。

### 4.3 调度系统需要同时解决的问题

无论是第一类还是第二类业务，调度框架都需要统一解决以下问题：
- 周期触发，例如每 3 分钟触发 A/B/C，每 5 分钟触发 D/E/F；
- 调度实例生成，避免漏跑、重跑和并发冲突；
- 选择执行模式：走“单应用多任务”还是“单任务单应用”；
- 在提交前生成或注入运行参数；
- 跟踪 Spark Application 运行状态；
- 失败重试、告警和状态记录。

## 5. 推荐的职责边界

在这个业务背景下，最合理的边界仍然是“外部总调度 + 项目内执行期调度”。

### 5.1 当前项目保留的能力

当前项目继续负责执行层能力：
- `--task-path` 单任务执行入口；
- `--app-config` 多任务宿主执行入口；
- 多任务模式下的任务发现、并发执行、失败隔离、自动关停；
- SQL 执行、UDF 注册、临时对象隔离、资源释放；
- 对调度系统传入的运行参数进行解析，并注入到任务配置或 SQL 上下文中。

当前项目不建议扩展为：
- 全局定时调度平台；
- 跨任务依赖编排中心；
- 全局运行实例管理中心；
- 多种引擎统一控制平面。

### 5.2 外部调度项目负责的能力

外部调度项目负责总调度和控制平面能力：
- 周期触发；
- 调度计划管理；
- 任务实例生成；
- 任务依赖关系解析；
- 运行模式选择；
- 参数组装与注入；
- `spark-submit` 提交；
- YARN 状态跟踪；
- 重试、告警、运行记录。

这意味着：
- 当前项目负责“一个任务怎么执行”；
- 外部调度项目负责“任务什么时候执行、按什么顺序执行、以什么参数执行”。

## 6. 两类业务如何落到调度框架

### 6.1 单进程多线程业务的调度方式

对于第一类业务，调度框架的行为应当是：
- 按周期触发一个宿主任务实例；
- 为该实例选定一个 `app-config.yaml`；
- 根据业务时间、批次号、环境参数生成一组运行参数；
- 通过 `spark-submit --app-config ...` 拉起一个 Spark Application；
- 由当前项目内部的 `TaskManager` 去发现并执行该 `app-config.yaml` 所关联目录下的所有 `sql.yaml`。

这里外部调度系统只负责“启动一次宿主任务”，不负责进入宿主进程内部再调度每个 `sql.yaml`。

这类业务推荐调度粒度是：
- 外部调度到“宿主应用实例”；
- 内部调度到“SQL 子任务”。

### 6.2 多进程编排业务的调度方式

对于第二类业务，调度框架的行为应当是：
- 维护一组有依赖关系的任务定义；
- 在一个调度周期内为这些任务生成实例；
- 按依赖关系决定提交顺序；
- 对每个任务单独组装 `config.yaml`、`sql.yaml` 路径和运行参数；
- 通过多次 `spark-submit --task-path ...` 或后续扩展的参数形式逐个拉起独立进程；
- 监听每个 Spark Application 的执行结果，再决定后续依赖任务是否可继续执行。

这类业务的核心不是应用内部并发，而是外部调度器对多个 Spark Application 的有序编排。

### 6.3 两类业务的统一抽象

建议外部调度项目把两类业务统一抽象为“调度任务定义 + 提交策略”：
- 调度任务定义：描述周期、依赖、参数模板、执行模式；
- 提交策略：决定最终用哪种 Spark 提交方式。

可抽象出两种提交策略：
- `HOST_APP`：对应单应用多任务模式，传入 `app-config.yaml`；
- `SINGLE_APP`：对应单任务单应用模式，传入 `task-path` 或等价配置。

这样外部调度项目不需要理解 Spark 内部实现细节，只需要根据任务定义选择合适的提交策略。

## 7. 参数注入设计建议

你提到的“在 spark-submit 时给不同进程注入一些参数，并能注入到对应的 sql.yaml 中”是非常合理的，这一层建议明确纳入调度设计。

### 7.1 参数注入的目标

参数注入建议解决以下问题：
- 不同批次下使用不同业务日期；
- 不同租户或环境使用不同路径、库名、分区值；
- 同一份 `sql.yaml` 在不同调度场景下复用；
- 避免为每个时间周期维护大量重复 SQL 配置。

### 7.2 推荐的注入位置

参数可以分两层：

第一层：调度层参数
- 例如 `biz_date`、`batch_time`、`run_id`、`env`、`region`；
- 由外部调度项目在实例生成时确定。

第二层：执行层参数
- 由当前项目在启动后接收这些参数；
- 再将参数注入：
  - `app-config.yaml`；
  - `config.yaml`；
  - `sql.yaml`；
  - 或 SQL 执行上下文。

### 7.3 推荐的注入方式

建议优先采用“模板变量替换”方式，而不是让调度层直接改写 SQL 文件本体。

例如：
- 调度层在提交时传入：
  - `--conf task.biz_date=2026-03-18`
  - `--conf task.region=cn`
- 执行层在读取 YAML/SQL 时解析 `${task.biz_date}`、`${task.region}` 这类占位符。

这样做的好处是：
- 调度层不需要直接修改文件；
- 执行层仍然掌握配置解析的一致性；
- 更容易支持两种执行模式共用同一套参数注入规则。

## 8. 周期调度与依赖调度如何共存

在你的场景里，周期调度和依赖调度不是二选一，而是需要同时支持。

### 8.1 周期调度

例如：
- 每 3 分钟触发一次 A、B、C；
- 每 5 分钟触发一次 D、E、F。

这部分由外部调度项目负责生成“某一时刻应该运行哪些任务实例”。

### 8.2 依赖调度

如果任务之间有依赖关系，例如：
- A 完成后才能跑 B；
- C 和 D 都完成后才能跑 E；

则外部调度项目需要在实例层面维护依赖关系，而不是只维护任务定义层面的依赖。

也就是说，要管理的是：
- “2026-03-18 10:00 这一批的 A 实例是否完成”；
- “只有这一批的 A 实例完成后，这一批的 B 实例才能提交”。

### 8.3 推荐做法

建议外部调度项目按“周期实例 + 依赖实例”统一建模：
- 周期规则负责产生日历时间上的任务实例；
- 依赖规则负责决定这些实例之间的提交顺序；
- 提交策略负责决定实例最终以哪种 Spark 模式执行。

## 9. 建议的大框架

建议整体分为三层。

### 9.1 配置层

负责维护：
- 单任务模式配置目录；
- 宿主模式 `app-config.yaml`；
- 任务元数据；
- 周期规则；
- 依赖规则；
- 参数模板。

### 9.2 调度层

建议新起一个独立调度项目，负责：
- 定时触发；
- 任务实例生成；
- 模式选择；
- 依赖判定；
- 参数渲染；
- `spark-submit` 提交；
- YARN 状态跟踪；
- 重试与告警。

### 9.3 执行层

当前 TaskPlugin 项目负责：
- 解析调度层传入的参数；
- 创建 SparkSession；
- 执行单任务或多任务；
- 在多任务模式下完成应用内轻量调度；
- 汇总状态并退出。

## 10. 一条可落地的实施路径

建议按以下顺序推进。

### 10.1 第一阶段

先做一个轻量外部调度项目，支持：
- Cron 周期触发；
- 两种提交模式选择；
- 基本的 `spark-submit` 封装；
- YARN 状态跟踪；
- 简单运行记录。

### 10.2 第二阶段

补齐参数注入能力：
- 调度层生成运行参数；
- 执行层支持参数解析；
- YAML 与 SQL 支持模板变量替换。

### 10.3 第三阶段

补齐多进程依赖编排：
- 定义任务依赖 DAG；
- 管理实例级依赖；
- 支持失败阻断、跳过、重试策略。

### 10.4 第四阶段

再逐步增强为更完整的调度平台：
- 告警；
- 人工补跑；
- 运行审计；
- Web 管理界面；
- 多引擎统一调度。

## 11. 最终结论

结合你的业务背景，推荐方案仍然是混合方案，但这次边界可以说得更明确：

- 当前项目内部负责执行期调度，重点承接“单进程多线程”的宿主式应用；
- 外部调度项目负责总调度，重点承接“多进程依赖编排”和“周期触发 + 参数注入”；
- 两类业务都由外部调度项目统一管理，但落地时选择不同的 Spark 提交策略。

如果压缩成一句话：
- 宿主式批量任务的子任务调度留在当前项目内；
- 多进程编排、周期触发、依赖控制、参数注入全部放到独立调度项目中。
