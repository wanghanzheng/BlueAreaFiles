# 大数据底座计算框架开发：GPT CodeX5.4 + Cursor + SDD

## 0写在前面

近年来，AI爆发式地飞速发展：从一个简单的ChatBot到音视图文多模态大模型，再到当今爆火的OpenClaw龙虾助手，只要给足权限和token，龙虾似乎能替我们完成80%以上的工作/生活琐事，谁又能想到08年钢铁侠里的贾维斯现在已经摆在了我们眼前。总的来说，AI的时代已经到来了，不谈未来，就是此刻，没人能够抗拒，与其固步自封，不如积极拥抱。



近期看到一张图，是一个研究报告，显示计算机科学领域是最高危受到AI冲击的领域之一，但至少我认为应当乐观地看待这个“冲击”。它可能是自软件工程诞生以来，推动这艘大船前进的最强引擎。当下多做尝试，才能从实践中找到通往更广阔海域的航向。

## 1开发背景

近期开发部UDA团队正在开发一个能够支撑上层大规模数据应用的大数据底座，为了提高开发效率和摸索AI coding的方式，部分工作我们尝试在蓝区做开发。我们尝试了部分编程工具（Claude Code，Cursor等）和部分大模型（GPT5.4，Gemini 3，Sonnet4.6等），在几周的亲身感受和实践后，大家不约而同地认为ChatGPT CodeX5.4 + Cursor是当前最合适的开发工具，加以SDD方法约束后产生的成果最符合预期。

下文将以一个数据计算的小需求做切入，总结一下这几周在蓝区开发的经验、遇到的问题和这套开发流程的优势，分享出来与大家交流讨论，欢迎大家勘误、指正。

### 1.1开发工具选择

单论AI编程工具的体验来说，Claude Code 和 Cursor 的界面差距并不大（都是在Vscode的基础上加入了一些对话框的UI），对于开发者来说都比较容易上手。只是归功于开发部大力支持了我们在蓝区的实践，给我们了一批Cursor的会员账户，每个月可以使用一定量token付费的模型（如Sonnet4.6）来辅助编程，因此就敲定一直用Cursor编程。但后续Cursor一些原生的问题开始显现：

账户数量有限，当一批人在使用时，容易出现其他人登录不上的情况。

每个月的token数量有限，几乎在月初就大规模地使用，很快就消耗殆尽。而Cursor免费的auto模型又幻觉严重，在处理项目方面表现较差。

Cursor本身提供的上下文窗口较小，虽然它官方文档说是动态调整的，但是使用感受确是容易丢失记忆。

正巧大家在交流群中对以上问题怨声载道，求解决方案时，OpenAI近期推出了GPT5.4。于是乎抱着试试的心态，尝试将模型换成了GPT5.4；幸运的是，GPT5.4的能力在实践中都得到了大家的认可，但token的问题依旧没有解决。直到有人发现了GPT本身的编程工具CodeX，它官方提供了对接Cursor的接口。

也就是说CodeX可以作为一个插件植入Cursor，你只要拥有一个GPT的会员账户，就可以在Cursor中直接使用CodeX来调用GPT5.4来替你管理项目、编写代码等，由于GPT的会员账户是按月度来算，并没有token限制，自此解决了token的问题，降低了开发成本。

### 1.2 浅析Vibe Coding与SDD编程的差异性

在AI编程前进道路上，经历了这样的阶段：1、传统编码：程序员自己根据需求文档自己设计代码，自己敲代码实现 2、AI辅助：程序员写占绝大篇幅的主逻辑，AI帮助补全一些零碎的部分。3、Vibe Coding：程序员只需说清楚需求、看结果、继续改就可以了，大量的代码细节交由AI产出。

自Vibe Coding出现以来，AI和程序员在开发中的主导地位就有些颠倒了，程序员接受了很多AI生成的代码，只要功能完善即可，程序员开始不再逐行审查所有的代码。这样做的优点很明显：上手快、原型快，甚至不是程序员都能做出自己的小程序并直接运行测试。但风险也很明显：代码质量、可维护性、安全性、可扩展性等等都是未知数。对于我们要做一个严肃、严谨的电信级的数据底座项目来说，Vibe Coding一定是不可取的。一位同事在闲谈时提到一句话，他说：“真正尝试去用AI做coding的人，对于某些小功能模块的开发可能会提速明显，但对于庞大复杂项目的落地，端到端功能的打通，AI coding能够提供的加速度应当是非常有限的。”我对此深以为然。

因此在实际开发应用中，在享受AI提速的同时，更要考虑如何给AI降速，这里就要提出所谓的SDD（Spec-Driven-Development 规格驱动开发）思想了。以最近比较火的SDD思想之一：OpenSpec为例。OpenSpec有一套OPSX扩展工作流命令: /opsx:new -> /opsx:ff -> /opsx:apply -> /opsx:verify -> /opsx:archive

## 2案例：计算框架新增SparkSQL读写StarRocks功能

当前计算框架的基本功能已经实现，能够读取相关配置和计算逻辑提交一个对应的Spark计算任务，而这个Spark任务就会对数据源进行读写操作。

之前只实现了计算框架提交的任务对接HDFS数据源的功能，新的需求要求我们能够实现计算框架提交的任务能够读写StarRocks里存放的数据。我打算以此需求为例，给大家分享这一套AI+SDD思想是如何帮助我们设计、开发、测试，最终闭环一个需求的全过程。

### 2.1 讨论、设计与决策

我认为很多人在借助AI时，会下意识地忽略SDD中的第一阶段，即创建需求单。大家很容易觉得某些需求就是一句话能说清楚的事情，比如就是把计算框架和StarRocks对接起来，听上去非常直白明确，于是就把这个需求喂给了AI，让AI替我们做设计。这样即使磕磕绊绊走到了最后，那我认为也只是实现了一个功能，而不是完成了一个需求。

在思考一个需求时，要同时从两方面去考虑，一个从使用者方面考虑，要想清楚我要什么；另一个从开发者方面考虑，要想清楚我要怎么做。比如我应当在设计开始前就想清楚，最终做完这个需求以后，用户会以什么形式指定一个计算任务，而用户又希望最终在StarRocks里得到什么样的结果。而我又应如何在保持现有代码架构不变的情况下，最小化的做修改。

有趣的是，这一部分其实也可以让AI参与进来，你可以把AI当作一个同事并且是相关领域软件专家，你简述一下需求内容，说说自己的想法，再让AI去思考以上两个问题，从而发表它的意见，你脑子里在与AI的对话讨论中，这个需求的目标和蓝图就会变得越来越清晰，至此才算完成了SDD中的第一阶段。

|  | 传统方式 | AI辅助 |
| --- | --- | --- |
| 做法 | 开发根据个人经验，结合当前项目结构和需求内容分析理解需求，对于不理解或没学过的需求有学习时间成本 | AI辅助指引、讨论；开发对于不理解的部分都可以直接询问AI |
| 耗时 | 0.5人/天~1人/天 | 1人/时~3人/时 |

明确需求后，就可以正式进入设计与决策阶段了（SDD的第二阶段）。

这一阶段我个人的习惯是让AI根据需求目标给我提2-3个方案，我会选择其中一个比较合适的再去打磨细节（如果没有合适的可以接着让AI提），在与AI的不断沟通中优化出一份最终方案，并让AI在这个项目的doc文件夹下维护一份需求文档.md文件，把这次沟通的最终方案完整地写下来。（需求文档的结构可以靠skill约束，见仁见智，无强制约束）

这里需要注意的是，在实践过程中，我认为到这一步为止，会过度放大与AI对话者本人的过往经验和思维方式。我司开发流程中推崇的“神三”流程中，有一步开发者反串讲，我认为就是一个很好的避免上述思维困局的方式。那么在AI辅助完成设计后，我会拉着SE和其他相关开发一起看一下最终的需求设计文档，集思广益才能发现设计的缺陷，修炼出更为上乘的“功法”。当然，最后可以把功法再传给AI，让它帮忙改需求文档落地，这一步的偷懒是合适且高效的。

举一个本次需求中真实出现的例子：最初我与AI敲定的设计文档里，设计了一个模块，能把配置文件中sql语句里的表名（逻辑表名）和真实StarRocks中的表名（物理表名）匹配起来，这样后续写sql语句的人就可以只用逻辑表名，无需关心真实物理表名。然而在后续与SE对过一遍之后，发现了这个模块非常冗余，因为在前期元模型的设计中，写sql语句的人他一定是能感知到StarRocks中的真实物理表名的，因此也不需要做什么匹配，他直接在sql语句中写真实物理表名即可。

|  | 传统方式 | AI辅助 |
| --- | --- | --- |
| 做法 | 开发设计<br>反串讲<br>开发修改方案 | AI设计<br>开发提要求<br>AI修改<br>反串讲<br>提出新的要求<br>AI修改 |
| 耗时 | 1人/天~2人/天 | 0.5人/天~1人每天 |

### 2.3 代码测试

其实关于单元测试，根据我的个人习惯已经在上一步代码生成时就顺带着完成了，目前唯一一次在实践中遇到AI写的代码报红，过不了单元测试的一次。究其原因也是我的java编译环境问题，是由于一些公司内部的依赖导致的，并不是代码本身的问题。

|  | 传统方式 | AI辅助 |
| --- | --- | --- |
| 做法 | 开发手写DT/UT | AI生成业务代码时连带生成测试代码 |
| 耗时 | 1人/天 | 几乎不额外耗时 |

至于端到端测试，事实上只要给足Agent足够的权限，提前在环境上搭好所有需要的数据源、数据库、各类依赖等等，AI就能自己完成端到端的测试闭环。但出于安全考虑，并不敢给AI开出这么高的权限，只是要求它完整地说明：做一次端到端的简单测试对环境有什么要求，具体是哪些步骤。最终具体的测试还是人去实现，只要提交任务到了计算框架，最终能在环境上的StarRocks看见写出来的数据即可。

### 2.4 总体效果

粗略估计，原本彻底闭环这个需求开发，大概需要4.5人/天；而真实情况是1个开发配合AI在1.5天内就可以完成，效率提升了66%。

## 3 关于AI Coding的一些感悟与思考

### 3.1 开发者个人身份的转变

AI coding 正在推动传统开发者的角色发生深层转变。开发者的价值，不再只体现在手工编写代码的速度、编写的代码质量上，而更深刻地体现在需求抽象、系统设计、边界定义与结果校验上。面对 AI，开发者需要站在更高一层，可以说要把自己当成一个SE或者MDE的角色，以架构者和设计者的视角进行交互，用SDD的思想，把业务目标、约束条件、实现路径和验收标准表达清楚。只有这样，AI 生成的内容才不会停留在“看似可用”，而是真正贴合需求预期、满足可维护性与可扩展性要求。未来的优秀开发者，将更像是能驾驭 AI 的系统设计者。

### 3.2 AI的能力边界控制与安全性

当前个人在AI coding的实践中，还是并不敢把full access的权限开放给AI。这就在其他人把命令输给AI以后，他去喝一杯咖啡的10-15分钟时间里，我仍旧得在电脑屏幕前时刻confirm窗口上不断弹出来的AI申请命令的权限。更劣势的是，其他人可以摒弃传统的调测模式，直接让AI去模拟或者构建所需的环境，编写相关脚本，直接在环境上运行起来。即使报错，AI也能根据错误日志或其他信息自我修复代码中的缺陷，直到脚本通过，符合预期为止，相当于AI自我闭环了整个调测+修改的过程，解放了人力。而我就只能古法炼丹，在AI和环境直接搬运信息，根据AI的指引做测试，实在是有种落后的感觉。

之所以选择古法炼丹，是认为AI的一系列操作毕竟是黑盒的，AI想要达成某个命令的目标，到底会怎么做，无人可知。在这过程中会不会破坏环境？也许生成代码时，生成的不符合预期了只需一个git restore就能回退掉，但是如果在环境上输入了破坏系统的命令，将会导致不可逆的后果。最近因为养龙虾导致龙虾不可控，从而造成了经济、名誉损失的案例不在少数。近期的一个访谈中，一个美国的AI安全专家表达了一种观点，AI既然能自己发现开发者难以发现的系统缺陷并修复，那么AI也可以用来攻击这些缺陷。综上所述，个人理解未来要做好AI权限的约束和AI能力的释放的平衡是一个宏大的课题，在保证安全性的情况下释放AI的最大能力将成为每个开发者的最大希冀。

### 3.3 AI Agent于个人和公司的影响

近来AI取代人类工作的论调甚嚣尘上，我个人理解AI是人类能力的放大器，而绝不是替代品，人类积累多年的软件工程经验和那些天马行空的想法是AI难以复制的。所以我更愿意把AI Agent看作新时代的软件杠杆。它放大的不是单纯的产出速度，而是人的认知、方法论和公司的组织协同能力。谁能更早理解这一点，谁就更有机会在这场变革中占据主动。AI时代真正稀缺的，不是会不会用AI，而是能否在AI参与的环境下，依然清晰定义问题、控制边界、保持创造力。真正驾驭AI的公司有能力把当初那些九天揽月的幻想真正落到实处，把公司内部的创新思想和技术实力真正转化为商业价值。
