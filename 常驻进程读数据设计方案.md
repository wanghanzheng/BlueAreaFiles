## 1 当前事实

当前存在多个 warehouse，每个 warehouse 在 HDFS 上都有独立根路径，例如：

```text
hdfs:///uda/warehouse1
hdfs:///uda/warehouse2
```

当前默认假设 warehouse 中存放的是 Parquet 文件。每个 warehouse 按业务时间组织目录，且每个 warehouse 有一个固定时间粒度 `grainType`，目前包括：

| grainType | 含义 | 示例目录 |
|---|---|---|
| `5m` | 5 分钟粒度 | `hdfs:///uda/warehouse1/20260416/12/00`、`.../12/05`、`.../12/10` |
| `15m` | 15 分钟粒度 | `hdfs:///uda/warehouse1/20260416/12/00`、`.../12/15`、`.../12/30` |
| `1h` | 1 小时粒度 | `hdfs:///uda/warehouse1/20260416/12/00`、`.../13/00` |
| `1d` | 1 天粒度 | `hdfs:///uda/warehouse1/20260416/00/00` |

常驻 polling 进程每轮需要解决三个问题：

1. 本轮应该读哪些输入数据。
2. 哪些输入已经被当前 `sql.yaml` 处理过，需要跳过。
3. 如何把本轮输入数据注册成业务 SQL 可访问的临时视图。

业务 SQL 最好只依赖一个统一入口，例如：

```sql
SELECT *
FROM ${tp.input.view}
```

路径、时间窗口、finished 状态、snapshot 进度等运行细节应尽量留在 Java 框架内部。

## 2 从前实现

从前实现的核心思路是：Java 侧根据时间窗口筛 HDFS 目录，再按目录打 finished 标记。

1. 根据 `grainType` 计算本轮检查范围。

   历史默认值如下，后续应允许配置覆盖：

   | grainType | 默认 startTime | 默认 endTime |
   |---|---|---|
   | `5m` / `15m` | 当前时间 - 6h | 当前时间 - 35m |
   | `1h` | 当前时间 - 24h | 当前时间 - 70m |
   | `1d` | 当前时间 - 72h | 当前时间 - 70m |

   示例：当前时间为 `2026-04-16 17:10`，warehouse 粒度为 `1h`，则本轮检查范围大致为：

   ```text
   2026-04-15 18:00 ~ 2026-04-16 16:00
   ```

   对应候选目录：

   ```text
   hdfs:///uda/warehouse1/20260415/18/00
   hdfs:///uda/warehouse1/20260415/19/00
   ...
   hdfs:///uda/warehouse1/20260416/16/00
   ```

2. 检查每个候选目录是否存在 finished 标记。

   如果某个目录已经有 finished 标记，则跳过；如果任务成功处理该目录，则在目录下写 finished 文件。

3. 将剩余路径放入 `List`，再按配置参数 `N` 切成多个 batch。

   每个 batch 最多包含 `N` 个目录。每个 batch 会使用DataSet合并读取这些目录下的 Parquet 文件，形成一个计算输入。

## 3 当前方案供选择

### 3.1 不引入 Iceberg 的方案

#### 3.1.1 保留原来输入文件夹路径的方案

该方案保持当前 HDFS 目录格式不变，仍然由 Java 侧生成本轮需要处理的路径。

处理流程：

1. Java 根据 `grainType + StartTime + EndTime` 计算本轮时间范围。
2. Java 按原目录格式拼出候选 HDFS 路径。
3. Java 检查路径是否存在，并过滤已经 finished 的路径。
4. 剩余路径按 `path-batch-size` 切成多个 batch。
5. 每个 batch 生成一个运行实例，并注册一个 `${tp.input.view}` 给业务 SQL 使用。

Spark SQL 示例：

```sql
CREATE OR REPLACE TEMPORARY VIEW polling_input_taskA_001
USING parquet
OPTIONS (
  path "hdfs:///uda/warehouse1/20260416/12/00,hdfs:///uda/warehouse1/20260416/13/00,hdfs:///uda/warehouse1/20260416/14/00"
);
```

业务 SQL 示例：

```sql
INSERT INTO target_table
SELECT *
FROM polling_input_taskA_001;
```

优点：

- 不需要改现有 HDFS 目录格式。
- Java 侧逻辑和旧实现接近，迁移成本低。
- finished 过滤后，Spark 只读需要处理的路径。

缺点：

- 多路径写入 SQL `OPTIONS(path "...")` 的兼容性需要验证。
- batch 很大时，`path` 字符串会很长。
- 目录格式不是 Hive-style，Spark SQL 无法自然基于分区列做裁剪。
- 如果改为 Java API `spark.read().load(pathsArray)` 会更稳，但就不再是纯 Spark SQL 拼接方式。

#### 3.1.2 修改原来输入文件夹路径的方案

该方案将 HDFS 目录改成 Hive-style 分区目录，使 Spark 能识别分区列。

目录标准：

```text
hdfs:///uda/<warehouse>/dt=yyyy-MM-dd/hour=HH/min=mm/
```

示例：

```text
hdfs:///uda/warehouse1/dt=2026-04-17/hour=00/min=00
hdfs:///uda/warehouse1/dt=2026-04-17/hour=01/min=00
hdfs:///uda/warehouse1/dt=2026-04-17/hour=02/min=00
```

处理流程：

1. Java 仍然根据时间范围生成候选分区。
2. Java 仍然负责检查目录存在性和 finished 状态。
3. 剩余分区按配置切成 batch。
4. 每个 batch 转换成 `dt/hour/min` 的 SQL 条件。
5. 框架先创建 warehouse raw view，再创建 batch input view。

Spark SQL 示例：

```sql
CREATE OR REPLACE TEMPORARY VIEW tp_raw_warehouse1
USING parquet
OPTIONS (
  path "hdfs:///uda/warehouse1"
);

CREATE OR REPLACE TEMPORARY VIEW polling_input_taskA_001 AS
SELECT *
FROM tp_raw_warehouse1
WHERE
  (dt = '2026-04-17' AND hour = '00' AND min = '00')
  OR (dt = '2026-04-17' AND hour = '01' AND min = '00')
  OR (dt = '2026-04-17' AND hour = '02' AND min = '00');
```

如果 batch 内时间连续，也可以生成更短的范围条件：

```sql
CREATE OR REPLACE TEMPORARY VIEW polling_input_taskA_002 AS
SELECT *
FROM tp_raw_warehouse1
WHERE dt = '2026-04-17'
  AND hour BETWEEN '00' AND '02'
  AND min = '00';
```

优点：

- 避免把大量路径拼进 `OPTIONS(path)`。
- 目录符合 Hive-style，Spark 更容易识别分区列。
- SQL 可读性比长路径列表更好。

缺点：

- 需要改造数据落地目录。
- Java 仍然要做候选分区生成、finished 过滤和 batch 切分。
- **如果 finished 后出现不连续时间点，`WHERE` 条件仍可能需要 `OR` 拼接**。
- 本质上仍是基于文件目录管理输入，不是表级元数据管理。

### 3.2 引入 Iceberg 管理数据而非直接管理 Parquet 路径的方案

引入 Iceberg 后，输入不再直接面向 HDFS 文件夹，而是面向 Iceberg 表。Iceberg 负责维护数据文件、快照、分区和元数据。Java 侧不需要再拼 HDFS 路径，也不需要按路径切 batch。

前提：

- 数据写入 Iceberg 表，而不是只把 Parquet 文件放到 HDFS 目录。
- 表中存在业务时间列，例如 `dataTime`。
- Spark 已配置 Iceberg catalog 和 Iceberg Spark SQL extensions。

#### 3.2.1 利用 Iceberg 的 `dataTime` 时间列筛选

该方案按业务时间处理数据。Java 侧仍然根据 `grainType + StartTime + EndTime` 计算本轮业务时间范围，但不再生成 HDFS 路径。

处理流程：

1. Java 计算本轮 `startTime` 和 `endTime`。
2. Java 判断当前 `taskId + timeRange` 是否已经 finished。
3. 若未完成，则为每个 `sql.yaml` 创建一个运行实例。
4. 框架创建 `${tp.input.view}`，SQL 中通过 `dataTime` 筛选本轮数据。
5. 业务 SQL 对 `${tp.input.view}` 做计算。
6. 成功后记录 `taskId + timeRange` finished。

Spark SQL 示例：

```sql
CREATE OR REPLACE TEMPORARY VIEW polling_input_taskA AS
SELECT *
FROM iceberg_catalog.dwd.warehouse1_table
WHERE dataTime >= TIMESTAMP '2026-04-17 00:00:00'
  AND dataTime <  TIMESTAMP '2026-04-17 12:00:00';
```

业务 SQL 示例：

```sql
INSERT OVERWRITE target_table
SELECT *
FROM polling_input_taskA;
```

说明：

- 时间条件推荐使用半开区间 `>= start AND < end`，避免相邻窗口边界重复。
- 如果表按 `dataTime` 做 Iceberg 分区，例如 `hours(dataTime)` 或 `days(dataTime)`，Iceberg 可以通过元数据裁剪不相关文件。
- 业务写入需要支持覆盖重写或幂等，因为失败重试时同一时间范围可能再次处理。

优点：

- 不再需要 HDFS 路径生成、路径 exists 检查、路径 batch 切分。
- 输入语义直接对应业务时间，容易理解和回补。
- **一个 `sql.yaml` 每轮通常只需要一个运行实例，相当于之前的方案一个sql.yaml要对应的子线程任务数与batch数量挂钩（几个batch的数据并发读取），当前的方案能产生最少的子线程。**

缺点：

- 仍然需要记录处理进度，例如 `taskId + timeRange`。
- 依赖 Iceberg 表对 `dataTime` 的合理分区和维护。

#### 3.2.2 利用 Iceberg 的 snapshot 机制筛选

该方案按 Iceberg 表提交进度处理数据，而不是按业务时间处理数据。

处理流程：

1. Java 查询当前 Iceberg 表最新 snapshot。
2. Java 从状态中读取当前 `taskId` 上次成功处理的 snapshot。
3. 框架创建当前 snapshot 范围对应的 changelog view 或增量输入 view。
4. 业务 SQL 处理该 snapshot 范围内的增量数据。
5. 成功后更新 `taskId -> lastProcessedSnapshotId`。

Spark SQL 示例：

```sql
CALL iceberg_catalog.system.create_changelog_view(
  table => 'dwd.warehouse1_table',
  options => map(
    'start-snapshot-id', '123456789',
    'end-snapshot-id', '223456789'
  ),
  changelog_view => 'polling_input_taskA'
);

SELECT *
FROM polling_input_taskA
WHERE _change_type = 'INSERT';
```

业务 SQL 示例：

```sql
INSERT INTO target_table
SELECT *
FROM polling_input_taskA
WHERE _change_type = 'INSERT';
```

说明：

- `start-snapshot-id` 表示上次成功处理的 snapshot，通常是 exclusive。
- `end-snapshot-id` 表示本轮要处理到的 snapshot，通常是 inclusive。
- 如果表存在 update/delete，需要明确业务是否处理 `_change_type` 中的 `DELETE`、`UPDATE_BEFORE`、`UPDATE_AFTER`。
- 如果只是 append-only 表，snapshot 增量处理会更简单。

优点：

- 进度状态非常简单：每个 `taskId` 只需要记录一个 `lastProcessedSnapshotId`。
- 能自然处理迟到数据，因为迟到数据只要是新提交，就会出现在新的 snapshot 增量中。
- 不依赖 `grainType`、HDFS 目录和业务时间窗口。

缺点：

- 语义是“提交增量”，不是“业务时间范围”。
- 一次 snapshot 可能包含多个 `dataTime` 范围的数据，业务 SQL 需要自行处理。
- update/delete、rewrite、compaction 等 Iceberg 操作会让增量语义变复杂。
- 如果目标表按业务时间覆盖写入，snapshot 方案不如 `dataTime` 方案直观。
- **严重缺陷：增量的数据处理会导致迟到数据也被处理，迟到数据应该丢弃**

### 3.3 初步建议

如果当前业务已经有稳定的 `dataTime` 字段，并且目标表支持按时间范围覆盖重写，优先推荐：

```text
3.2.1 Iceberg + dataTime 时间窗口方案
```

原因：

- 它能去掉 HDFS 路径筛选和路径 batch 切分。
- 它保留当前 polling 的业务时间窗口语义。
- 一个 `sql.yaml` 每轮通常只生成一个运行实例。
- Spark SQL 表达简单，核心就是 `dataTime >= start AND dataTime < end`。

如果暂时不能引入 Iceberg，则在 3.1 中选择：

- 目录不能改：选 3.1.1。
- 目录可以改且希望 SQL 分区裁剪更自然：选 3.1.2。

如果未来业务更偏向“消费每次表提交的增量”，而不是按业务时间窗口覆盖计算，再考虑 3.2.2 snapshot 方案。



## 4 IceBerg读数据与Spark的关系

是的，**有关系，但不是简单的“BETWEEN 范围越大，executor 越多就一定越快”**。

可以这样理解：

```
SELECT * FROM iceberg_table WHERE dataTime >= TIMESTAMP '2026-04-17 00:00:00'  AND dataTime <  TIMESTAMP '2026-04-18 00:00:00' 
```

这条 SQL 交给 Spark 后，大致会发生几件事：

1. Iceberg 根据 dataTime 条件做元数据裁剪。

   如果表按 dataTime 分区，或者文件级统计信息比较好，Iceberg 会先排除不相关的数据文件。

2. Spark 拿到需要读取的数据文件列表。

   比如这个 BETWEEN 最终命中：

   `500 个 Parquet data files `

3. Spark 把这些文件拆成 scan tasks / partitions。

   这些 task 会分配给 executor 并发读取。

4. executor 并行执行 task。

   executor 越多、core 越多，理论并发读取能力越强。

所以 BETWEEN 划出的数据量越大，通常会命中更多文件、产生更多 Spark tasks，也就更能利用更多 executor 并发处理。

但并发能力取决于几个因素：

**1. 数据文件数量和大小**

如果 BETWEEN 只命中 3 个大文件，即使你有 100 个 executor，也不一定能充分并发。

如果命中 1000 个合理大小的文件，就更容易并发。

所以 Iceberg/Parquet 文件大小很重要。一般太大的文件会降低并行度，太多小文件会带来调度和元数据开销。

**2. Spark executor 数量和 core 数量**

并发任务数大致和总 core 数相关：

```
总并发 task 数 ≈ executor 数 × 每个 executor 的 cores 
```

例如：

```
10 个 executor，每个 4 core => 理论上同时跑约 40 个 task 
```

如果这次查询只有 20 个 scan task，40 个 core 也用不满。

如果有 1000 个 scan task，40 个 core 就会分批处理。

**3. Iceberg 的文件规划策略**

Iceberg 会根据文件大小、split size 等参数规划扫描任务。实际 Spark task 数不完全等于文件数。

相关参数可能包括：

```
read.split.target-size read.split.open-file-cost 
```

它们会影响一个 scan task 读多少数据。

**4. 后续 SQL 是否有 shuffle**

读数据只是第一步。如果业务 SQL 里有：

```
GROUP BY JOIN DISTINCT ORDER BY WINDOW 
```

后续会发生 shuffle。此时性能还取决于：

```
spark.sql.shuffle.partitions executor memory 网络 IO 数据倾斜 
```

不只是输入读取并发。

**5. 集群资源和 HDFS/对象存储带宽**

executor 多了以后，底层存储也要扛得住并发读。否则 executor 增加到一定程度后，瓶颈会变成 HDFS 带宽、NameNode 元数据压力或磁盘 IO。

所以结论是：

**BETWEEN 决定本次读多少数据；Iceberg/Spark 会把命中的数据文件规划成并发任务；executor 数量决定这些任务能以多大并发度执行。**

更直白一点：

```
BETWEEN 范围越大 -> 通常命中文件越多 -> 可并行任务越多 executor/core 越多 -> 同时处理任务能力越强 
```

但是否更快，还要看：

```
命中文件是否足够多 文件大小是否合理 是否有 shuffle 集群 IO 是否够 executor 内存是否够 
```

如果你准备让一个常驻进程每轮用一个较大的 BETWEEN 读很多数据，那么确实需要根据数据量配置更合适的 executor 数量、executor cores、shuffle partitions。
不过在 Spark 里，**更常调的是总 core、文件大小、shuffle partitions，而不是单纯 executor 个数**。
