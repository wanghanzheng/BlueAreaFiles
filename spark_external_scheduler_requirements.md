# Spark 调度需求记录

维护日期：2026-06-09

## 1. 背景

另一个工程项目名为 `HDP`，后续 Spark 任务的启停和调度逻辑会在 `HDP` 项目中实现。

当前先在本文档中记录 Spark 调度相关需求，等 `HDP` 工程项目后续搬到本地后，再结合实际代码继续细化和实现。

## 2. 元模型中的 Spark 调度类型

元模型目前将 Spark 任务定义为三种调度类型：

1. 事件触发
2. 每天定时触发
3. 常驻隔一段时间触发

映射到当前 Spark 计算框架：

- 事件触发：对应 `SINGLE` 模式。
- 每天定时触发：对应 `SINGLE` 模式。
- 常驻隔一段时间触发：对应 `POLLING` 模式。

## 3. HDP 中已有的任务启动函数

`HDP` 项目中已经存在任务启动函数：

```java
asynchExecuteTask(taskName, taskHdfsPath)
```

该函数的语义：

- 入参只需要提供任务名和任务配置所在的 HDFS 路径。
- 函数内部负责启动对应 Spark 任务。
- 该函数不区分具体执行模式，可以启动 `SINGLE` 模式任务，也可以启动其他模式任务。

## 4. 任务发现与内存实例维护

### 4.1 总体思路

`HDP` 需要维护后面两类调度任务的内存类实例：

- 每天定时触发任务：`daily_trigger`
- 常驻隔一段时间触发任务：`interval_trigger`

当元模型在 HDFS 中新增、修改或删除任务目录时，`HDP` 需要能感知到这些变化，并同步更新内存中的任务类实例。

为避免把职责混在已有调度进程里，需要新增一个专门的 HDFS 任务发现同步进程。该进程只负责扫描 HDFS，并维护内存中的任务类实例，不直接负责按时间触发任务，也不直接负责常驻任务重启。

### 4.2 任务注册表

`HDP` 内部建议维护一个全局任务注册表，用于保存已经被发现并实例化的任务。

注册表 key 可以采用“调度类型 + 任务名”的形式，例如：

```text
daily_trigger:taskC
interval_trigger:taskC
```

注册表 value 是对应的任务类实例。

### 4.3 扫描范围

HDFS 任务发现同步进程需要扫描以下固定根目录：

```text
hdfs://hacluster/UDA/daily_trigger/*
hdfs://hacluster/UDA/interval_trigger/*
```

事件触发任务由 MML 命令直接启动，暂时不纳入该内存实例同步机制。

### 4.4 扫描频率

建议扫描频率：

- `HDP` 启动时立即全量扫描一次。
- `HDP` 运行期间每 1 分钟扫描一次。

选择 1 分钟的原因：

- 每天定时触发进程本身也是每分钟检查一次是否到点。
- 任务发现同步进程和定时触发粒度保持一致，可以减少新增任务刚好错过当天触发时间的问题。
- 扫描逻辑可以做成轻量扫描，避免每分钟完整解析所有配置。

扫描频率后续可以做成可配置项。如果任务量特别大，后续可以根据 HDFS 压力调整为更长间隔。

### 4.5 同步规则

每次扫描后，HDFS 任务发现同步进程需要将 HDFS 上的任务目录和内存任务注册表做对比。

#### 4.5.1 新增任务

如果 HDFS 上存在某个任务目录，但内存注册表中没有对应实例，说明元模型新增了任务。

处理方式：

1. 解析该任务目录下的配置。
2. 创建对应任务类实例。
3. 将实例加入内存任务注册表。
4. 后续由对应调度进程管理该任务。

例如：

```text
元模型新增 hdfs://hacluster/UDA/daily_trigger/taskC
        ↓
HDP 的 HDFS 任务发现同步进程扫描发现 taskC
        ↓
解析 taskC 配置
        ↓
创建 daily_trigger:taskC 类实例
        ↓
放入任务注册表
        ↓
每天定时触发进程后续可以感知并管理 taskC
```

如果新增的是：

```text
hdfs://hacluster/UDA/interval_trigger/taskC
```

则创建 `interval_trigger:taskC` 类实例，后续由常驻任务管理进程感知并管理。

#### 4.5.2 已有任务配置变更

如果 HDFS 上存在某个任务目录，内存注册表中也存在对应实例，则认为是已有任务。

处理方式：

1. 检查任务目录或配置文件是否发生变化。
2. 如果配置发生变化，则更新内存中对应任务类实例的属性。

对于 `daily_trigger` 任务：

- 启动时间等属性可以更新到内存实例中。
- 具体每天启动时间的配置刷新逻辑仍然由原有的每天定时触发 HDFS 配置同步进程负责。
- 本任务发现同步进程只负责保证任务实例存在，以及同步基础实例信息。

对于 `interval_trigger` 任务：

- 如果 `POLLING` 任务配置发生变化，更新内存中的配置或待生效配置。
- 如果该任务当前正在 YARN 上运行，不需要立即重启。
- 等任务达到最大运行周期退出后，由常驻任务管理进程在下个周期重新启动，此时启用新配置。

#### 4.5.3 删除任务

如果内存注册表中存在某个任务实例，但 HDFS 上已经不存在对应任务目录，说明元模型删除了该任务。

处理方式：

1. 判断该任务是否仍在 YARN 上运行。
2. 如果任务仍在运行，则直接 kill 掉对应 YARN Application。
3. 从内存任务注册表中移除该任务类实例。
4. 后续不再管理和触发该任务。

### 4.6 与其他进程的职责边界

HDFS 任务发现同步进程只负责“发现任务”和“维护内存实例”：

- 发现 HDFS 新增任务，并创建类实例。
- 发现 HDFS 删除任务，并 kill 运行中的任务后移除类实例。
- 发现基础配置变化，并更新类实例属性或待生效配置。

它不负责：

- 按每天几点触发任务。
- 每小时刷新每天定时触发任务的启动时间。
- 定期检查 `POLLING` 任务是否达到最大运行周期退出。
- 从 YARN 定期同步 `POLLING` 任务运行状态。

这些职责仍然由对应的调度进程或监控进程完成。

## 5. 事件触发调度需求

### 5.1 任务配置存放方式

事件触发类 Spark 任务属于 `SINGLE` 模式。

这类任务的配置文件固定存放在 HDFS 路径下，包括：

- `app-config.yaml`
- `config.yaml`
- `sql.yaml`

示例路径：

```text
hdfs://hacluster/UDA/event_trigger/taskA
hdfs://hacluster/UDA/event_trigger/taskB
```

可以理解为每个事件触发任务都有一个固定的 HDFS 任务目录。

### 5.2 MML 命令控制

事件触发任务由 MML 命令控制。

MML 命令在 `HDP` 项目中后续也会体现为一个函数调用，并携带任务名。

示例：

```text
StartTask(taskA)
```

含义：

- MML 命令指定要启动哪个任务。
- `HDP` 接收到命令后，根据任务名找到对应任务。
- 调用已有的 `asynchExecuteTask` 函数启动一次该任务。

### 5.3 HDP 侧处理逻辑

事件触发调度在 `HDP` 侧的核心逻辑是：

1. 接收 MML 命令。
2. 从 MML 命令中解析任务名。
3. 根据任务名获取该任务固定的 HDFS 配置路径。
4. 调用 `asynchExecuteTask(taskName, taskHdfsPath)`。
5. 启动一次对应的 `SINGLE` 模式 Spark 任务。

### 5.4 当前理解

事件触发调度本身不需要复杂的周期计算、依赖编排或常驻进程管理。

它的重点是建立：

- MML 命令到任务名的映射。
- 任务名到 HDFS 任务目录的映射。
- MML 命令到 `asynchExecuteTask` 的调用链路。

## 6. 每天定时触发调度需求

### 6.1 任务配置存放方式

每天定时触发类 Spark 任务属于 `SINGLE` 模式。

这类任务的配置文件在 HDFS 上也有固定路径，目录下同样存放任务运行需要的配置文件，例如：

- `app-config.yaml`
- `config.yaml`
- `sql.yaml`

示例路径：

```text
hdfs://hacluster/UDA/daily_trigger/taskA
hdfs://hacluster/UDA/daily_trigger/taskB
```

### 6.2 HDP 中的任务类

每天定时触发任务在 `HDP` 中需要有对应的类实例表示。

该类至少包含以下属性：

- 每天几点启动。
- 任务名。
- 任务路径。

可以理解为：每个每天定时触发任务，在 `HDP` 内部都有一个常驻的任务定义对象，由该对象保存任务的触发时间和 HDFS 配置路径。

### 6.3 看时间的调度进程

`HDP` 中需要有一个负责看时间的进程，用来管理所有每天定时触发任务的类实例。

该进程的行为：

1. 同时管理所有每天定时触发任务的类实例。
2. 每分钟刷新一次。
3. 检查每个任务是否已经到达当天配置的启动时间。
4. 如果某个任务到时间了，则调用：

```java
asynchExecuteTask(taskName, taskHdfsPath)
```

5. 启动一次对应的 `SINGLE` 模式 Spark 任务。

### 6.4 扫 HDFS 的配置同步进程

除了看时间的调度进程，还需要一个扫 HDFS 的进程，用来同步每天定时触发任务的配置变化。

该进程的行为：

1. 每 1 小时扫描一次 HDFS。
2. 扫描每天定时触发类任务的固定配置路径。
3. 读取任务配置中的每天启动时间。
4. 如果发现 HDFS 配置中的每天启动时间发生变化，则更新 `HDP` 中对应任务类实例的属性。

这个进程的目的不是直接启动任务，而是让 `HDP` 内存中的任务类实例和 HDFS 上的配置保持一致。

### 6.5 当前理解

每天定时触发调度由两个后台进程配合完成：

- 看时间的进程：每分钟检查一次任务是否到点，到点后调用 `asynchExecuteTask` 启动任务。
- 扫 HDFS 的进程：每 1 小时检查一次任务配置是否变化，如果启动时间变了，就更新内存中的任务类实例。

这类任务本身仍然通过 `asynchExecuteTask(taskName, taskHdfsPath)` 启动，和事件触发任务一样，最终启动的是一次 `SINGLE` 模式 Spark 任务。

## 7. 常驻隔一段时间触发调度需求

### 7.1 任务配置存放方式

常驻隔一段时间触发类 Spark 任务对应当前计算框架中的 `POLLING` 模式。

这类任务的配置文件在 HDFS 上也有固定路径，目录下存放任务运行需要的配置文件，例如：

- `app-config.yaml`
- `config.yaml`
- `sql.yaml`

示例路径：

```text
hdfs://hacluster/UDA/interval_trigger/taskA
hdfs://hacluster/UDA/interval_trigger/taskB
```

### 7.2 HDP 中的任务类

为了方便检查任务是否正在运行，需要为这类常驻任务在 `HDP` 中也定义对应的类实例。

该类至少包含以下属性：

- 任务名。
- 任务路径。
- 运行状态。

其中运行状态由监控进程从 YARN 查询后更新，供管理进程判断是否需要重新启动任务。

### 7.3 常驻任务管理进程

`HDP` 中需要有一个统一管理 `POLLING` 常驻任务的进程。

该进程的行为：

1. 在 `HDP` 进程启动时，加载所有常驻隔一段时间触发类任务。
2. 将所有这类任务都启动到 YARN 上常驻运行。
3. 启动任务时仍然调用已有函数：

```java
asynchExecuteTask(taskName, taskHdfsPath)
```

4. 每隔 1 小时检查一次自己管理的常驻任务。
5. 如果发现有任务已经因为达到最大运行周期而退出，则统一重新启动该任务。

### 7.4 YARN 状态监控进程

为了让常驻任务管理进程方便判断任务状态，需要增加一个监控进程，定期从 YARN 查询这类任务的运行状态。

该进程的行为：

1. 每半小时查询一次 YARN。
2. 获取所有常驻隔一段时间触发类任务的运行状态。
3. 将查询到的状态更新到 `HDP` 中对应任务类实例的运行状态属性上。

这个监控进程不直接负责启动任务，只负责同步 YARN 侧的真实运行状态。

### 7.5 当前理解

常驻隔一段时间触发调度由两个后台进程配合完成：

- 常驻任务管理进程：在 `HDP` 启动时统一拉起所有 `POLLING` 任务，并每隔 1 小时检查是否有任务退出；如果有任务退出，则重新调用 `asynchExecuteTask` 启动。
- YARN 状态监控进程：每半小时从 YARN 查询任务运行状态，并更新到内存中的任务类实例上。

这类任务和前两类任务不同，不是每次触发就启动一个短生命周期 `SINGLE` Application，而是启动到 YARN 上后保持常驻运行；当任务达到最大运行周期退出后，再由管理进程负责重新启动。
